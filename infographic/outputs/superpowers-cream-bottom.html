TDD 强制流程

Test-Driven Development
RED
失败测试

先写 failing 测试，验证测试正确失败，确认测试覆盖了预期行为

GREEN
快速通过

实现让测试通过的代码，最小化改动，不追求完美只追求通过

REFACTOR
重构优化

重构代码结构，删除任何为让测试通过而临时添加的代码，确保干净

关键约束：代理在 REFACTOR 阶段会自动删除任何为让测试通过而临时添加的代码，确保代码库始终干净

触发短语

Trigger Phrases
可直接在会话中使用的触发语
"我要实现 X 功能，需求是…先帮我做需求澄清并给出设计方案"
"帮我把这个设计分成小任务并给出每个任务的文件变更和测试步骤"
"现在开始执行计划，从第一个任务开始"
"为刚写的功能生成失败测试，然后实现并修复直到测试通过"
"请对当前分支进行代码审查并列出严重问题"
"完成当前开发分支并给我可选的合并策略"
"帮我做 brainstorm" / "触发 brainstorming"

常见问题

FAQ
如何手动触发某个 skill？
直接在会话里说"帮我做 brainstorm"或"帮我写计划"，代理会在合适时机自动选择对应技能
插件如何更新？
/plugin update superpowers gemini extensions update
子代理写出坏代码怎么办？
两阶段审查机制会拦截——规范合规检查 + 代码质量审查，严重问题阻止任务推进
测试基线不干净怎么办？
使用 using-git-worktrees 创建隔离工作树，确保每次开发都在干净测试基线上开始

最佳实践

Best Practices
✓
设计优先：在任何自动化写代码之前，通过 brainstorming 获取清晰设计并确认，避免返工
✓
小步提交：每个任务 2-5 分钟，频繁自测，利用 git worktrees 保持主分支干净
✓
明确验收标准：每个任务必须有明确的测试/脚本/手动验证步骤，不能含糊
✓
严重性分类：CRITICAL/HIGH 必须修复才能继续，MEDIUM 可延后，LOW 可选
✓
两阶段审查：子代理每次提交前先自测 spec 合规性，再做代码质量审查
