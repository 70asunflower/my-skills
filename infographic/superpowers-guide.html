03 推荐使用流程

1

启动 Brainstorming

用自然语言描述需求，代理通过提问澄清需求，分段展示设计文档并等待确认

"我要实现X功能，需求是…先帮我做需求澄清并给出设计方案"
2

审核并批准设计

阅读代理输出的分段设计，逐段确认或请求修改，确认后再继续

3

生成实现计划 Writing Plans

设计分解为 2-5 分钟的小任务，每个任务包含：目标描述 / 文件路径与变更 / 完整代码片段 / 验证步骤

4

创建隔离工作区 Using Git Worktrees

在新分支创建隔离工作树，运行安装和测试命令，确保干净的测试基线

5

执行计划 Subagent-Driven Development

每个任务启动子代理 → 子代理按规范自测(spec compliance) → 代码质量审查 → 提交并运行测试

6

测试驱动开发 TDD

始终遵循 RED → GREEN → REFACTOR 循环：写失败测试 → 实现代码 → 检查通过 → 重构并清理临时代码

7

代码审查 Requesting / Receiving Code Review

关键节点生成审查报告，按严重性分类（CRITICAL / HIGH / MEDIUM / LOW），严重问题阻塞任务推进

8

完成分支 Finishing Development Branch

所有任务完成且测试通过后，提供选项：直接合并 / 发起 PR / 保留分支 / 放弃并清理工作树

04 技能库

🤝 brainstorming

编码前的设计与提问环节，分块展示设计方案并等待用户确认，确保理解一致后再开始实现

🌿 using-git-worktrees

创建隔离工作树，在新分支上设置项目并验证测试基线，保持主分支干净不受未完成工作污染

📋 writing-plans

将设计拆解为 2-5 分钟的小任务，输出每个任务的文件路径、完整代码变更和验证步骤

⚡ subagent-driven-development

为每个任务启动子代理执行，进行两阶段审核（先对照规范，再检查代码质量），小步提交频繁自测

🧪 test-driven-development

强制 RED-GREEN-REFACTOR 循环：写失败测试 → 实现通过 → 重构，并移除任何临时/多余代码

🔍 requesting-code-review

在关键节点生成代码审查报告，识别严重问题并阻止进度直到修复，同时记录普通问题供后续处理

📥 receiving-code-review

接收并处理来自其他代理或人类的审查反馈，按严重性排序并逐项解决，直到所有关键问题关闭

🎯 finishing-a-development-branch

验证测试通过，提供最终选项：直接合并、发 PR、保留分支继续开发、或放弃并清理工作树

06 TDD 强制流程

RED

先写 failing 测试
验证测试正确失败

GREEN

实现让测试通过
最小化代码改动

REFACTOR

重构代码结构
删除临时代码

关键约束：代理在 REFACTOR 阶段会自动删除任何为让测试通过而临时添加的代码，确保代码库始终干净、无多余逻辑积累。

07 最佳实践

✓
设计优先：在任何自动化写代码之前，通过 brainstorming 获取清晰设计并确认，避免返工
✓
小步提交：每个任务 2-5 分钟，频繁自测，利用 git worktrees 保持主分支干净
✓
明确验收标准：每个任务必须有明确的测试/脚本/手动验证步骤，不能含糊
✓
严重性分类：CRITICAL/HIGH 必须修复才能继续，MEDIUM 可延后，LOW 可选
✓
两阶段审查：子代理每次提交前先自测 spec 合规性，再做代码质量审查

08 常见问题

❓

如何手动触发某个 skill？

直接在会话里说"帮我做 brainstorming"或"帮我写计划"，代理会在合适时机自动选择对应技能

🔄

插件如何更新？

Claude: /plugin update superpowers
Gemini: gemini extensions update superpowers

🐛

子代理写出坏代码怎么办？

两阶段审查机制会拦截——规范合规检查 + 代码质量审查，严重问题阻止任务推进并标记待修复

🧪

测试基线不干净怎么办？

使用 using-git-worktrees 创建隔离工作树，确保每次开发都在干净测试基线上开始
